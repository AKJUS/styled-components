# React Native CSS Features Compatibility

This is a living index of modern CSS capabilities and their current state on the React Native path. Features are grouped into what works today, what we polyfill, and what still needs upstream React Native support to become usable.

The goal is transparency: if you rely on something in this document, you can see whether it runs today, how we handle it under the hood, and what would need to change upstream to move a "not supported" row into the "polyfilled" or "native" column.

If a feature you need isn't listed, or the workaround section needs updating, open a comment or pull request to extend this file.

## Status snapshot

As of React Native 0.85:

- Many modern CSS features work natively, including CSS string transforms, box-shadow with spread/inset, filter chains, gradients, slash-alpha colors, `hwb()`, `mix-blend-mode`, `isolation`, `cursor`, `display: contents`, logical margin/padding/inset longhands, and several logical border longhands.
- Many features that React Native does not parse natively are polyfilled by styled-components: either converted before styles reach React Native or resolved against the live device environment.
- Other features still require upstream React Native work to support meaningfully. This document is a contract with the community: when one of those ships, we wire it up and move the row out of the unsupported section.

## Works natively in React Native 0.85

- `aspect-ratio`, `backface-visibility`, `border-radius`, `box-sizing`, `display: contents`, `object-fit`, `opacity`, `outline-offset`, `overflow`.
- `background-image: linear-gradient()` / `radial-gradient()` + `background-size` / `background-position` / `background-repeat`.
- `box-shadow` with offset, blur, spread, inset, and multi-shadow lists.
- Colors: hex, `rgb()` / `rgba()`, `rgb(r g b / a)` slash-alpha, `hsl()` / `hsla()`, `hwb()`, named colors, `transparent`.
- `cursor`, `direction`, `mix-blend-mode`, `pointer-events`, `user-select`.
- `isolation: isolate` scopes descendant `mix-blend-mode` to the element's own compositing group on every target. Android has no isolation style key, so styled-components also lifts React Native's hardware-texture layer prop to create the isolated surface; without it, blended descendants composite against the page background there.
- `filter: blur() brightness() contrast() drop-shadow() grayscale() hue-rotate() invert() opacity() saturate() sepia()` chains. On iOS, RN 0.85 requires the SwiftUI filter backend (`ReactNativeReleaseLevel` experimental) for `blur`, `saturate`, `hue-rotate`, `grayscale`, `contrast`, and `drop-shadow`; `brightness` and `opacity` work without that flag. Caution: while a gated filter function is mounted on iOS at the default release level, descendants' `mix-blend-mode` also stops compositing against the backdrop (device-verified); development builds warn once when this applies.
- Flex: `flex-grow` / `flex-shrink` / `flex-basis`, `align-items`, `justify-content`, `align-content`.
- Gap: `gap` / `row-gap` / `column-gap`.
- Logical margin / padding / inset longhands in the inline and block axes, plus their `-top` / `-bottom` / `-left` / `-right` counterparts. React Native also supports logical border colors/radii and start/end widths; styled-components fills the remaining `border-inline-*` / `border-block-*` CSS syntax below.
- Position: `absolute` / `relative`, plus `static` on the new renderer.
- `transform: translateX(10px) rotate(45deg) scale(1.2) matrix(<9 or 16 numbers>)` as a CSS string or transform array. React Native 0.85 accepts `matrix` arrays of 9 or 16 numbers, but the CSS string parser does not support the `matrix3d()` function name, and non-zero translate lengths need units. On Android, `skewX()` / `skewY()` parse but React Native drops the rendered skew; development builds warn once.

## Polyfilled by styled-components

These work in your style templates even though React Native doesn't parse them natively. styled-components either rewrites the value before handing it to React Native, or re-evaluates it on each render against the live environment.

When `styled-components/native` is bundled for `react-native-web`, the browser handles CSS surfaces it already supports where react-native-web can pass them through safely. That means `light-dark()`, viewport units including `dvh` / `svh` / `lvh` / `vi` / `vb`, and mixed-unit `calc()` / `min()` / `max()` / `clamp()` stay as CSS instead of being approximated through the iOS / Android polyfills. Static modern color functions are converted to sRGB hex before they reach react-native-web, because React Native's color normalization otherwise drops them before the browser can render them.

### React Native pseudo-states (equivalent to CSS pseudo-classes)

Interaction pseudo-states resolve through the live state React Native provides, which only `Pressable` (or a custom component forwarding its style to one) supplies. On other elements (`View`, `Text`, etc.) the base styles still apply but the pseudo buckets stay inert with a one-time development warning, since those hosts have no press / hover / focus state to track. A bucket fenced behind a non-matching media query (e.g. web-only `@media (hover: hover)` rules) stays silent.

| CSS                      | React Native mapping                                                                             |
| ------------------------ | ------------------------------------------------------------------------------------------------ |
| `&:active` / `&:pressed` | `Pressable` pressed state.                                                                       |
| `&:disabled`             | Component `disabled` prop.                                                                       |
| `&:focus` / `&:focused`  | `onFocus`/`onBlur` on focusable inputs.                                                          |
| `&:hover`                | `Pressable` hover state (desktop / web) or `onHoverIn`/`onHoverOut` (iOS pencil / desktop-like). |

### Image and ImageBackground

- **`Image`**: authored layout and paint (`width`, `height`, `margin`, `opacity`, `borderRadius`, `tintColor`, `resizeMode` / `objectFit`, filters and shadows on the view) map to RN `Image` / wrapper styles like any other view-backed component.
- **`ImageBackground`**: the outer container accepts normal **view** styles (positioning, flex, padding). The bitmap layer uses React Native’s **`imageStyle`** (or equivalent) for `resizeMode`, `borderRadius` on the image, `tintColor`, and background image sizing; layered CSS `background` from templates may split between **container** and **image** styles after compilation. Prefer explicit layout on the wrapper when you need predictable clipping or padding around the image.

### TextInput, Switch, and Button (style vs props)

- **`TextInput`**: treat **box model**, **typography**, **colors**, **outline**, and **opacity** as normal template CSS when they map to RN style keys. **Placeholder text and color**, **caret / selection colors**, **keyboard type**, **secure text entry**, **return key behavior**, **autocorrect**, **multiline** scrolling, and similar controls are **React props** (or platform-specific props), not something to spell as non-standard CSS in templates. Some real CSS (for example `caret-color`) may map to RN props where the platform exposes them; see the polyfilled and native rows above when listed.
- **`Switch`**: **track and thumb colors**, **disabled** state, and iOS background tint are **props**. Layout around the control (margins, flex alignment) is **style** on the same component or a parent `View`.
- **`Button`**: RN’s `Button` is not a general-purpose styled surface. For CSS-shaped controls, compose **`Pressable` (or `Touchable*`) plus `Text`** so template CSS applies to ordinary view and text children.

## Routing values onto wrapped components with `attrs`

Some CSS properties only target a specific React Native primitive on native: `text-overflow` and the line-clamp surface drive props on `<Text>`; `accent-color` drives `<Switch>`; `overscroll-behavior`, `scrollbar-width`, and `scroll-snap-type` drive `<ScrollView>` props. The default polyfill lifts the right props automatically when you style one of those primitives directly.

When you wrap a third-party component instead (e.g. a Slider, Checkbox, ProgressBar, BottomSheet), you can route the resolved CSS value onto whatever prop the wrapped component expects using the function form of `.attrs(...)` with the AST bridge. The second argument exposes `peek(key)` (read) and `pop(key)` (read and consume).

```tsx
import Slider from '@react-native-community/slider';
import styled from 'styled-components/native';

const ThemedSlider = styled(Slider).attrs<{ thumbTintColor?: string }>((_props, ast) => ({
  thumbTintColor: ast.pop('accentColor') as string | undefined,
}))`
  accent-color: red;
`;
```

`pop('accentColor')` returns the resolved value (a string, a theme token, or a `PlatformColor` for `accent-color: auto`) and removes it from the style bag so it doesn't end up as an unrecognized style key on the wrapped component. Use `peek` if you also need the value to stay in the style. The same pattern works with any CSS property: write the standards-compliant declaration in the template, route the value onto whichever prop the wrapped component actually reads.
